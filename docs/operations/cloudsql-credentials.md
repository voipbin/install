# Cloud SQL credentials

Reference for VoIPBin's Cloud SQL user inventory, password rotation, and operator-side recovery flows.

## 1. Inventory

`voipbin-install apply` creates the following Cloud SQL users.

| User | Host | DB scope | Notes |
| --- | --- | --- | --- |
| `bin-manager` | `%` | MySQL `bin_manager` | Primary application user. |
| `asterisk` | `%` | MySQL `asterisk` | registrar-manager + Kamailio realtime tables. |
| `call-manager` | `%` | MySQL `bin_manager` | call-manager binary. |
| `kamailioro` | `10.0.0.0/255.0.0.0` | MySQL `asterisk` (read-only) | Network-restricted; see §2. |
| Postgres `bin-manager` | n/a | Postgres `bin_manager` | rag-manager (pgvector). |

Passwords are generated by terraform `random_password` resources with `override_special = "!*+-._~"` (URL-safe RFC 3986 userinfo-unreserved + sub-delim subset).

## 2. kamailioro host pin and registry deferral

The kamailioro user is pinned to host `10.0.0.0/255.0.0.0` so only callers from the `10.0.0.0/8` network can authenticate. This matches the network shape Kamailio uses inside the VPC.

The kamailioro user is intentionally excluded from the `scripts/terraform_reconcile.py` registry. The Cloud SQL provider's import id format is `{project}/{instance}/{host}/{name}` (4 slash-delimited fields); the host literal contains a slash, breaking the importer (`google_sql_user.resourceSqlUserImporter` v5.45.2 onward). Consequence. terraform creates the user on first apply. If state is lost AND the user remains in Cloud SQL, subsequent `terraform apply` returns "user already exists" and the operator must manually delete-and-recreate (see §5).

Remote terraform state (PR-K) makes state loss rare in practice.

## 3. Password generation and URL encoding

Locked alphabet. `A-Za-z0-9!*+-._~`. Length 24. Generated by `random_password.mysql_kamailioro`.

The Kamailio `KAMAILIO_AUTH_DB_URL` is built by `scripts/ansible_runner.py::_build_kamailio_auth_db_url`.

1. Re-validate password against the locked alphabet (regex gate). RuntimeError if widened.
2. Splat raw password into `mysql://kamailioro:<raw>@<mysql-private-ip>:3306/asterisk`. **No percent-encoding.** Kamailio's `db_mysql` driver (libmysqlclient) does NOT percent-decode the password component of the URL; the password bytes pass through verbatim to MySQL. Dogfood iter#11 (2026-05-13) verified this live: raw password authenticates, percent-encoded form is rejected with `Access denied`.

What happens if alphabet drifts. helper raises `RuntimeError` referencing this doc; `voipbin-install apply` fails visibly. Fix path. update terraform `override_special` AND extend the helper's alphabet regex AND extend tests AND confirm the new characters do not collide with URL-structural delimiters (`:`, `/`, `@`, `?`, `#`, space, `%`) when emitted raw.

## 4. Rotation procedure

```bash
# 1. Replace the random_password resource so terraform regenerates the
#    Cloud SQL user password. Run from the installer's terraform working
#    directory so the correct backend/state config is loaded. If your normal
#    apply path passes -backend-config=... flags, pass the SAME flags here.
cd terraform/
terraform init -reconfigure
# Sanity check the backend landed against the right state before destructive ops:
terraform state list | grep random_password.mysql_kamailioro || {
  echo "ERROR: state does not contain random_password.mysql_kamailioro"; exit 1; }
terraform apply -replace=random_password.mysql_kamailioro
cd -

# 2. Run the FULL install pipeline so terraform_reconcile updates config
#    (cloudsql_private_ip is read-only here; nothing changes) and ansible
#    re-templates .env with the new password. Do NOT run --stage ansible_run
#    alone; see §6 for why.
voipbin-install apply
```

Downtime. Brief Kamailio service interruption while the kamailio play's restart handler cycles both VMs (default `kamailio_count = 2`, no `serial: 1` on the play today). Customer-visible impact. PSTN registration via auth_db fails between password replacement and `.env` re-template, typically under 60 seconds.

To minimize the window. either schedule during low-traffic, or coordinate a planned drain (stop Kamailio first, then steps 1 and 2 in sequence, then start Kamailio).

## 5. Manual rescue (state loss with surviving user)

If terraform state is lost (e.g., state bucket corruption) and the kamailioro user still exists in Cloud SQL.

```bash
# Confirm the user exists.
gcloud sql users list --instance=<instance>

# Delete it. This triggers an immediate auth_db outage on Kamailio until
# step 2 completes; expect a brief auth window where auth_db queries fail
# before _build_kamailio_auth_db_url re-renders .env. PJSIP signaling
# (non-DB) is unaffected.
gcloud sql users delete kamailioro --host=10.0.0.0/255.0.0.0 --instance=<instance>

# Re-apply. Terraform recreates the user; reconcile populates config;
# ansible re-templates .env.
voipbin-install apply
```

## 6. Standalone `--stage ansible_run` hazard

Running `voipbin-install apply --stage ansible_run` without a preceding `--stage terraform_reconcile_outputs` (or full apply) means the helper sources `cloudsql_private_ip` from whatever config-on-disk happens to contain. On a fresh workstation or after a config reset, this is empty, and the helper returns `""`. Ansible then re-templates `.env` with an empty `KAMAILIO_AUTH_DB_URL`, silently breaking Kamailio auth_db on the next container restart.

Mitigation. always run the full pipeline (`voipbin-install apply`) for rotation or `.env` re-templates. If a staged invocation is necessary, prefix with `--stage terraform_reconcile_outputs --stage ansible_run` to refresh config first. PR-E's `check_cloudsql_private_ip` preflight (`scripts/preflight.py::check_cloudsql_private_ip`) already rejects empty `cloudsql_private_ip` at full-pipeline apply time.

## 7. Brownfield pre-install check

If voipbin is being installed onto a project where a kamailioro user already exists from a prior hand-managed setup.

```bash
gcloud sql users list --instance=<instance> | grep kamailioro || echo "absent (good)"
```

If the user is present. options.

1. Drop it. `gcloud sql users delete kamailioro --host=10.0.0.0/255.0.0.0 --instance=<instance>`. Then run `voipbin-install apply`.
2. Reset state and recreate. terraform import via the 4-part id is unsupported for hosts containing slashes, and `random_password.result` is a computed attribute that cannot be externally assigned. The correct sequence is. (a) `terraform state rm random_password.mysql_kamailioro` (b) `terraform state rm google_sql_user.voipbin_mysql_kamailioro` (c) `gcloud sql users delete kamailioro --host=10.0.0.0/255.0.0.0 --instance=<instance>` (d) `voipbin-install apply`. Functionally equivalent to option 1 for greenfield installs; prefer option 1 for simplicity.

## 8. Other credentials

The Postgres `bin-manager` password and the Postgres install are documented in PR-D1's design doc (`docs/plans/2026-05-12-pr-d1-cloudsql-postgres-design.md`). MySQL `bin-manager` / `asterisk` / `call-manager` user passwords share the same locked alphabet and reach Pods via `PLACEHOLDER_DSN_PASSWORD_*` substitution into a DSN string that is consumed by Go's `database/sql` MySQL driver, which expects raw passwords. The kamailioro flow also emits raw passwords (see §3 above); the original PR-D2c design assumed Kamailio's `db_mysql` would percent-decode the URL, which dogfood iter#11 (2026-05-13) disproved — all MySQL consumers in this codebase now agree on raw-password semantics.
