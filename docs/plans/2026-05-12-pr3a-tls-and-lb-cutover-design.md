## 1. Goal

Single PR that moves TLS termination from cert-manager/ingress to a
**Pod-level self-signed cert** auto-generated by the installer, and
simultaneously moves external traffic from nginx-ingress to **5
per-Service GCP LoadBalancers** with the static IPs reserved by PR #10.

After this PR merges, a fresh install runs end-to-end:
1. Operator runs `terraform apply` → 5 static IPs reserved (5 names
   stay reserved; only 4 are wired to Services in PR #3a — the
   hook-manager IP sits unused until PR #3b).
2. Operator runs `init` → self-signed cert generated, Secrets created,
   manifests applied, 4 LBs come up bound to the api/admin/talk/meet
   reserved IPs.
3. Operator creates **4** DNS A records (api/admin/talk/meet)
   pointing at the LB IPs. **hook A record waits for PR #3b**
   because hook-manager Pod has no 80/443 listener configured in this
   PR — exposing the LB now would either route nothing (if no ports
   defined) or leak the 2112 metrics port externally.
4. Browser/webhook traffic reaches Pods directly via LB, Pod-side TLS
   (api-manager only). Frontends served as plain HTTP at LB 80 →
   Pod 80 in PR #3a; PR #3b adds the nginx sidecar for TLS.
5. Operator replaces the self-signed cert with a real cert before
   production (PR #3c documents this).

PR #3b adds the `square-manager` namespace, the frontend rename to
`admin/talk/meet`, the nginx sidecar for frontend TLS, the
`*-internal` ClusterIP split for metrics/audiosocket, and the
`hook-manager` LoadBalancer Service flip (now that Pod gains 80/443
containerPorts and NetworkPolicy is in place). PR #3c does
config-schema cleanup, Terraform variable updates, and README rewrite.

### Decision lineage

## 5. TLS bootstrap (the heart of this PR)

### 5.1 Module: `scripts/tls_bootstrap.py`

```python
def bootstrap_voipbin_tls_secret(
    namespace: str = "bin-manager",
    hostnames: list[str] | None = None,
    secret_name: str = "voipbin-tls",
    valid_days: int = 3650,
) -> str:
    """
    Idempotent. If the Secret already exists in `namespace`, returns
    "exists" and DOES NOT overwrite. Otherwise generates a 10-year
    self-signed RSA-2048 cert with the given SAN list and creates
    the Secret with two data keys:

      tls.crt       PEM-encoded cert (single cert, no chain)
      tls.key       PEM-encoded private key (PKCS8)

    Returns "created" on first run.

    Private key is generated in memory via `cryptography.hazmat`; never
    written to operator filesystem. Secret upload to Kubernetes uses
    `kubectl apply -f -` via stdin pipe — no tempfile.

    Default hostnames if None:
      ["api.<PLACEHOLDER_DOMAIN>", "hook.<PLACEHOLDER_DOMAIN>",
       "admin.<PLACEHOLDER_DOMAIN>", "talk.<PLACEHOLDER_DOMAIN>",
       "meet.<PLACEHOLDER_DOMAIN>"]
    All 5 hosts are included in the SAN list even though admin/talk/meet
    don't yet receive Pod-level TLS in PR #3a. This is forward-looking
    for PR #3b's nginx sidecar so the same cert covers all frontends
    without regeneration. CommonName is set to "voipbin-self-signed"
    (a placeholder string), since modern TLS clients honor SAN over
    CN; tests assert SAN list match exactly.
    The placeholder is substituted to the operator's real domain by
    the caller before this function runs.
    """
```

### 5.2 `bin-manager` Secret content alignment

`bin-api-manager` and `bin-hook-manager` Go code reads:
- `SSL_CERT_BASE64`: base64-encoded PEM cert.
- `SSL_PRIVKEY_BASE64`: base64-encoded PEM private key.

**Verified 2026-05-12** against monorepo:
- `bin-api-manager/internal/config/main.go:103-104` viper bindings.
- `bin-hook-manager/internal/config/config.go:53-54` viper bindings.
- `bin-api-manager/k8s/deployment.yml:68,73` and
  `bin-hook-manager/k8s/deployment.yml:56,61` envFrom references.

Both services share **the same single env-var pair** — there is no
`_API_` or `_HOOK_` suffix. The install repo's `voipbin-secret` is
mounted into both Pods via `envFrom: secretRef: voipbin-secret`,
exposing one cert key + one privkey key to both.

`voipbin-tls` is a separate Kubernetes TLS Secret with
`tls.crt`/`tls.key` data fields (kubectl-native TLS Secret shape). It
serves future consumers (PR #3b nginx sidecar for frontends). The
Pod-level Go code does not read it directly.

Therefore `bootstrap_voipbin_tls_secret` will:
1. Generate cert + key once.
2. Write `voipbin-tls` Secret (`tls.crt`, `tls.key` data fields) in
   `bin-manager` via `kubectl apply -f -` stdin pipe. This Secret is
   newly created (does not exist on fresh install), so apply has no
   prior keys to merge.
3. Patch `voipbin-secret` Secret with **exactly two keys**:
   `SSL_CERT_BASE64` and `SSL_PRIVKEY_BASE64`, base64-encoded. Use
   `kubectl patch -n bin-manager secret voipbin-secret --type=merge -p
   '{"data":{"SSL_CERT_BASE64":"<b64>","SSL_PRIVKEY_BASE64":"<b64>"}}'`.
   The `--type=merge` JSON-merge-patch preserves the 6 existing
   non-SSL keys (JWT_KEY, DB_USER, DB_PASSWORD, REDIS_PASSWORD,
   RABBITMQ_PASSWORD, API_SIGNING_KEY). No risk of accidental
   deletion that `kubectl apply` 3-way merge could pose.

### 5.2.1 Idempotent + atomic-pair contract

Bootstrap protects against three operator scenarios:

| Initial state of voipbin-secret SSL keys | Bootstrap action |
|---|---|
| Both keys absent or empty, and `voipbin-tls` Secret missing | Generate cert. Patch `voipbin-secret` FIRST (atomicity anchor), then create `voipbin-tls` Secret with the same cert/key. |
| Both keys absent or empty, but `voipbin-tls` Secret already exists (recovery from a prior partial run) | Delete the stale `voipbin-tls` first, then take the fresh-generate path above. This guarantees both Secrets always hold the SAME cert pair, preventing divergence that would break PR #3b's nginx sidecar. |
| Both keys non-empty (operator pre-supplied real cert) | Skip `voipbin-secret` patch AND skip `voipbin-tls` creation. Log: "SSL keys already populated; `voipbin-tls` Secret will be created by operator or PR #3b consumer." This avoids the inconsistency of having `voipbin-secret` hold a real cert while `voipbin-tls` holds a different self-signed cert. PR #3b's frontend sidecar gracefully handles missing `voipbin-tls` Secret (deferred to that PR's design). |
| Exactly one key non-empty (partial fill) | Hard error. Log "SSL_CERT_BASE64 and SSL_PRIVKEY_BASE64 are partially set — refusing to mix bootstrap key with operator-provided value. Fix manually." Abort `init`. |

**Atomicity ordering rationale.** The two kubectl writes (patch
`voipbin-secret`, apply `voipbin-tls`) are not atomic as a pair. Order
the patch BEFORE the apply so that:
- Patch fails → `voipbin-tls` not created → retry takes fresh-generate
  path cleanly.
- Patch succeeds, apply fails → `voipbin-secret` holds the cert →
  retry hits the both-populated skip branch (row 3 above). Operator
  is told to manually create `voipbin-tls` or wait for PR #3b's
  sidecar consumer to surface the NotFound.

In neither failure mode do the two Secrets diverge.

Rationale for the change vs prior iteration: PR #3a does not yet have
a consumer of `voipbin-tls` Secret (sidecar comes in PR #3b). Creating
the Secret only when bootstrap generates fresh cert keeps the trust
domain singular (one cert source per install run). Operator who
pre-supplies real cert into `voipbin-secret` will hit a clear "Secret
not found" error if PR #3b sidecar consumer runs before they create
`voipbin-tls`; PR #3b design adds explicit guidance.

This contract is enforced in code and asserted by unit tests in §9.

**Option B (rejected):** change `bin-api-manager` and `bin-hook-manager`
to read TLS files from a mounted volume. Rejected because it requires
a monorepo PR before PR #3a can merge, breaking the decision to keep
this PR self-contained.

## 6. Manifest changes

### 6.1 `scripts/k8s.py` — add 5 ADDRESS tokens

5 new substitution entries `PLACEHOLDER_STATIC_IP_ADDRESS_*` fed from
Terraform `*_static_ip_address` outputs.

Safe default value: empty string. Rationale: an empty
`spec.loadBalancerIP:` is the K8s convention for "let the controller
pick any address", so the manifest stays valid under `kubectl apply
--dry-run=server` but does NOT bind a reserved IP if the operator
forgets to wire Terraform output. Operator notices via `kubectl get
svc` showing an unexpected IP. Acceptable failure mode for fresh
install.

`PLACEHOLDER_STATIC_IP_NAME_*` tokens stay (PR #10 API preservation;
unused by manifests in this PR).

### 6.2 `k8s/backend/services/api-manager.yaml`

External Service spec replaces `ClusterIP`. **Only port 443 is
exposed externally; metrics (2112) and audiosocket (9000) are
intentionally OMITTED from this Service** to avoid leaking internal
endpoints on the public LB IP:

```yaml
spec:
  type: LoadBalancer
  loadBalancerIP: PLACEHOLDER_STATIC_IP_ADDRESS_API_MANAGER
  externalTrafficPolicy: Cluster
  ports:
    - name: https
      port: 443
      targetPort: 443
  selector:
    app: api-manager
```

**Consequence (transitional):** internal callers that reach
`api-manager:2112` (Prometheus scrape) and `api-manager:9000`
(audiosocket from `voip` namespace Asterisk Pods deployed by this same
install repo at `k8s/voip/asterisk-call/`, `asterisk-conference/`,
`asterisk-registrar/`) **lose the cluster-DNS endpoint**
`api-manager.bin-manager.svc.cluster.local:9000` until PR #3b creates
`api-manager-internal` ClusterIP Service.

Audiosocket traffic flow: `voip` namespace Asterisk Pods stream AI
audio bytes over TCP 9000 to api-manager. The endpoint is removed
between PR #3a merge and PR #3b merge. Fresh installer who completes
PR #3a but has not yet wired a SIP carrier will not generate
audiosocket traffic, so the gap is invisible — but the moment the
operator completes carrier provisioning before PR #3b lands,
audiosocket calls will silently fail. **Acceptable per pchero's
transitional rule, but tracked in PR #3b acceptance as the highest
unblocking priority among #3b items.** PR #3b design doc must
explicitly call out `api-manager-internal` creation as gating any
production-readiness signaling.

Deployment unchanged. Pod still reads `SSL_CERT_BASE64` /
`SSL_PRIVKEY_BASE64` from `voipbin-secret`; PR #3a's bootstrap fills
those values.

### 6.3 `k8s/backend/services/hook-manager.yaml`

**PR #3a leaves `hook-manager` Service as ClusterIP (unchanged from
main).** The LoadBalancer flip is deferred to PR #3b for these
reasons:

1. The Pod has no 80/443 listener configured in `install/k8s/backend/
   services/hook-manager.yaml` (Deployment containerPort only lists
   2112 metrics). PR #3b adds the 80/443 containerPorts.
2. Without 80/443 listeners, a hook-manager LB would either expose
   port 2112 (Prometheus metrics) externally — a security exposure —
   or have an invalid empty `ports` array.
3. NetworkPolicy `allow-ingress-to-hook` (required for external
   ingress) is also deferred to PR #3b.

The reserved static IP `PLACEHOLDER_STATIC_IP_NAME_HOOK_MANAGER`
remains reserved by Terraform (PR #10) but is not consumed by any
Service in PR #3a. Operator's DNS A record for `hook.<domain>`
should be created in PR #3b, not in PR #3a.

### 6.4 Frontend Services (`square-admin.yaml`, `square-talk.yaml`, `square-meet.yaml`)

For each:
```yaml
spec:
  type: LoadBalancer
  loadBalancerIP: PLACEHOLDER_STATIC_IP_ADDRESS_<ADMIN|TALK|MEET>
  externalTrafficPolicy: Cluster
  ports:
    - name: http
      port: 80
      targetPort: 80
  selector:
    app: square-<admin|talk|meet>
```

Namespace stays `bin-manager`. Service/Deployment name stays `square-*`.
Plain HTTP at LB 80 → Pod 80. PR #3b adds Pod-level TLS via nginx
sidecar and moves to `square-manager` ns with rename.

**Browser-experience caveat (corrected from iter-1):** mixed-content
is **not** the issue between PR #3a and PR #3b. Modern browsers
permit a plain-HTTP origin (`http://admin.<domain>`) to call HTTPS
APIs (`https://api.<domain>`). The actual blocker is that
`api-manager:443` serves a **self-signed cert** that browsers reject
with `NET::ERR_CERT_AUTHORITY_INVALID` until the operator (a) visits
`https://api.<domain>` directly and accepts the cert, or (b) replaces
the self-signed cert with a CA-issued one. Documented in §7 risk #5.
PR #3c README covers this in the Production Checklist.

### 6.5 NetworkPolicy

No changes in PR #3a. `allow-ingress-to-api` (443+9000 unrestricted)
and `allow-ingress-to-frontends` (80 unrestricted) already permit
external LB traffic to the relevant Pods. `hook-manager` LB cannot
reach Pod regardless of NetworkPolicy because Pod has no listener;
PR #3b adds both Pod listener AND `allow-ingress-to-hook`.
